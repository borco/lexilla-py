## What this is

A permissively-licensed (MIT) Python binding for
[Lexilla](https://www.scintilla.org/Lexilla.html), the lexer library used by
Scintilla since 5.0. Built with nanobind + CMake + scikit-build-core. Lexilla
itself has no Qt dependency, so this package has none either — it just
creates `ILexer5` lexer instances that can be handed to *any* Scintilla
binding via `SCI_SETILEXER`, not only [pyside6-scintilla](https://github.com/borco/pyside6-scintilla)
(a sibling project, same author).

## What this is NOT

- Not a Scintilla binding itself — pairs with one (e.g. pyside6-scintilla)
- Not affiliated with the Scintilla/Lexilla project
- Not a redesign of Lexilla's API — exposes `CreateLexer`/`ILexer5` as-is

## Stack

- **Language**: C++ (binding glue) + Python (package/API)
- **Binding generator**: nanobind
- **Build system**: CMake + scikit-build-core
- **Package manager**: uv
- **CI**: GitHub Actions + cibuildwheel
- **Lexilla**: vendored release tarball, not a live hg clone, under
  `src/lexilla_vendor/`. A few Scintilla interface headers it needs but
  doesn't ship (`ILexer.h`, `Sci_Position.h`, `Scintilla.h`) are separately
  vendored under `src/scintilla_interface/` — see `docs/auditing.md`.

## Dev machines

Linux x86_64/aarch64, Windows x86_64, macOS arm64 + x86_64

## Conventions

- Python package name on PyPI: `lexilla`
- Import name: `lexilla`
- Versioning: follow Lexilla's version (e.g. Lexilla `5.5.x` → this package
  `5.5.x.N`)
- All new code under MIT license; vendored Lexilla code retains HPND license
- Repo is named `lexilla-py` (not `lexilla`); within it, the vendored
  upstream source lives at `src/lexilla_vendor/` (not `src/lexilla/`) so it
  never collides with the Python package directory of the same likely name
  on case-insensitive filesystems
- Cross-binding integration (e.g. handing a created lexer to
  pyside6-scintilla's `SCI_SETILEXER`) goes through the lexer's raw pointer
  exposed as a plain `int` by default — zero coupling between packages. The
  `lexilla[pyside6-scintilla]` extra adds optional convenience glue that
  imports pyside6-scintilla directly.

## Python style

For hand-written Python (anything not generated by nanobind/CMake):

- Prefer bare `Final` over a parameterized `Final[T]` when the type is
  inferable, for module-level constants and attributes that are set once
  and never reassigned.
- Names are either public (`name`) or private (`__name`, name-mangled) —
  no single-underscore "protected" convention.
- Module-level functions are usually public.
- Add brief docstrings — including for private methods/attributes —
  explaining the *why* and *what*. Keep them short; don't restate what a
  well-named identifier already says.
