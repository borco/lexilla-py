# CLAUDE.md

This file grounds Claude sessions working on this repository.
Read `docs/specs/mission.md` for full background. Key facts:

## What this is

A permissively-licensed (MIT) Python binding for
[Lexilla](https://www.scintilla.org/Lexilla.html), the lexer library used by
Scintilla since 5.0. Built with nanobind + CMake + scikit-build-core. Lexilla
itself has no Qt dependency, so this package has none either. It is kept as
a separate package from [pyside6-scintilla](https://github.com/borco/pyside6-scintilla)
(a sibling project, same author) — in practice its only consumer — so this
binding's release cadence and vendored Lexilla version can track upstream
Lexilla releases independently of pyside6-scintilla's own release cycle.

## What this is NOT

- Not a Scintilla binding itself — pairs with one (e.g. pyside6-scintilla)
- Not affiliated with the Scintilla/Lexilla project
- Not a redesign of Lexilla's API — exposes `CreateLexer`/`ILexer5` as-is

## Stack

- **Language**: C++ (binding glue) + Python (package/API)
- **Binding generator**: nanobind
- **Build system**: CMake + scikit-build-core
- **Package manager**: uv
- **CI**: GitHub Actions + cibuildwheel
- **Lexilla**: vendored release tarball, not a live hg clone, under
  `src/lexilla_vendor/`. A few Scintilla interface headers it needs but
  doesn't ship (`ILexer.h`, `Sci_Position.h`, `Scintilla.h`) are separately
  vendored under `src/scintilla_interface/` — see `docs/auditing.md`.

## Dev machines

- Windows 11 + WSL (Debian/Ubuntu) — primary
- macOS M1 Mac Mini — secondary

## Platforms targeted

Linux x86_64/aarch64, Windows x86_64, macOS arm64 + x86_64

## Conventions

- Python package name on PyPI: `lexilla`
- Import name: `lexilla`
- Versioning: follow Lexilla's version (e.g. Lexilla `5.5.x` → this package
  `5.5.x.N`)
- All new code under MIT license; vendored Lexilla code retains HPND license
- Repo is named `lexilla-py` (not `lexilla`); within it, the vendored
  upstream source lives at `src/lexilla_vendor/` (not `src/lexilla/`) so it
  never collides with the Python package directory of the same likely name
  on case-insensitive filesystems
- Cross-binding integration (e.g. handing a created lexer to
  pyside6-scintilla's `SCI_SETILEXER`) goes through the lexer's raw pointer
  exposed as a plain `int` by default — zero coupling between packages.
  Convenience glue that imports pyside6-scintilla directly is also provided
  for ergonomics.

## Python style

For hand-written Python (anything not generated by nanobind/CMake):

- Prefer bare `Final` over a parameterized `Final[T]` when the type is
  inferable, for module-level constants and attributes that are set once
  and never reassigned.
- Names are either public (`name`) or private (`__name`, name-mangled) —
  no single-underscore "protected" convention.
- Module-level functions are usually public.
- Add brief docstrings — including for private methods/attributes —
  explaining the *why* and *what*. Keep them short; don't restate what a
  well-named identifier already says.

## Where to find things

- `src/lexilla_vendor/` — vendored Lexilla source
- `src/scintilla_interface/` — vendored Scintilla headers Lexilla needs but

## Roadmap

The [project board](https://github.com/users/borco/projects/3) is the joint
roadmap for this repo and
[pyside6-scintilla](https://github.com/borco/pyside6-scintilla) (sibling
project, same author). Issues/PRs from both repos are auto-added by
`.github/workflows/add-to-project.yml`.

When asked what to work on next, check the project board's "In progress"
lane for an item belonging to this repo (`borco/lexilla-py`). If one exists,
work on it. If there is no item for this repo in "In progress", say so and
stop — don't guess or pull something from Backlog instead.

If this repo has a `continue-prompt.md`, its "Next up" section should point
to the board's "In progress" lane rather than naming a specific issue (so it
can't go stale), while its "Current state" section keeps the detailed,
issue-specific work notes the board can't hold (blockers, decisions, file
paths). Before trusting "Current state", confirm it still describes whatever
issue the board currently shows as "In progress" — if the board has moved on
(or shows nothing), treat the old notes as stale.

## Git

- Never run `git commit` (or `git push`) unless explicitly told to. Stage
  and present changes for review first.
- When starting work on a tracked issue, branch off the *current* branch
  (not always `master`, which may not hold relevant in-progress work) as
  `issue/NN/short-description` (e.g. `issue/9/bind-named-styles`), and
  prefix commit subject lines with `ref #NN:` (e.g. `ref #9: Bind
  NamedStyles/NameOfStyle`), where `NN` is this repo's issue number.
- These branches merge back with `--no-ff`, preserving their commit
  history as a group on the parent branch. Before merging, check whether
  the parent has moved since branching and flag it -- `--no-ff` doesn't
  replace rebasing onto an advanced parent. Ask before rebasing; don't do
  it automatically.
